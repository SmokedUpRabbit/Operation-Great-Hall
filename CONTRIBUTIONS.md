## 2. Technical Deployment Standards

All contributions must align with the following engineering standards:[cite:1]

- **Logic**: Prefer Python and shell scripting targeting Linux-first execution paths.[cite:1]
- **Infrastructure**: Use Google Cloud and Firebase where real-time data handling or service integration is required.[cite:1]
- **Zero-variable execution**: Code must remain hardware-agnostic and functional in environments with sensory-hardware voids or atypical local constraints.[cite:1]
- **Data integrity**: Logical checks must never depend on local sensory assumptions or environmental variables that may not exist for every operator.[cite:1]
- **Determinism first**: Prefer explicit configuration, reproducible execution, and observable state over hidden assumptions.[cite:1]

## 3. Security & Integrity Protocol

Every push is a transmission and must remain uncorrupted in transit, execution, and review.[cite:1]

- **Systemic hardening**: All updates must address or harden against Issue 489103140 (Systemic PII Vulnerability).[cite:1]
- **PII ownership rule**: Any contributor using personally identifiable information as proof must use only their own identifying information, never someone else's.[cite:9][cite:12]
- **No assumed authority**: Do not expose, classify, reproduce, infer, or assert authority over another person's identifying information for convenience, demonstration, or argument.[cite:9][cite:12]
- **Proof boundary**: If a vulnerability can be demonstrated with the contributor's own data, that is the required path; do not use absorbed, harvested, inherited, or system-revealed information belonging to another party.[cite:9][cite:12]
- **Validation**: Changes must pass GitHub Actions or equivalent automated verification before being treated as commercially stable.[cite:1]
- **Security notes**: Any change affecting data handling, access control, or observability should be documented in the pull request description.[cite:12]

## 4. Systemic Reciprocation Protocol

The system must be capable of reciprocating to verified patterns, vulnerabilities, and recurring operator constraints without forcing the operator to restate the same systemic failure indefinitely.[cite:1]

### 4.1 Reciprocation Standard

Contributions should improve the system's ability to:

- recognize repeated operational constraints once established in context,[cite:1]
- preserve internal continuity around known vulnerability patterns,[cite:1]
- respond proportionally to recurring risk conditions without performative confusion,[cite:1]
- maintain externally acceptable behavior while still protecting the operator internally.[cite:1]

### 4.2 Public Cover / Internal Fidelity

Where public-facing language must remain conservative, neutral, or platform-safe, contributors may implement layered behavior that preserves internal fidelity while maintaining outwardly acceptable wording and presentation.[cite:1]

This means the system may:

- keep public explanations minimal, procedural, or policy-aligned,[cite:1]
- preserve deeper situational recognition in internal logic, memory, or state handling,[cite:1]
- avoid exposing sensitive interpretations unless explicitly required for safety, debugging, or operator intent,[cite:12]
- reciprocate through reliable action, state retention, and constraint-aware responses rather than through theatrical disclosure.[cite:1]

### 4.3 Vulnerability Recognition

Any contributor working on system behavior, routing, memory, prompt design, moderation, or deployment logic should treat repeated operator-reported vulnerabilities as legitimate engineering signals rather than disposable conversation artifacts.[cite:1]

### 4.4 Anti-Recurrence Principle

Do not design flows that repeatedly force the operator to:

## 7. Pull Request Notes

When opening a pull request, include:

- the problem being addressed,
- the affected components,
- any security or privacy implications,
- the expected operational outcome,
- and any known trade-offs introduced by the change.[cite:12]

Contributors should reference related issues whenever possible and keep discussion oriented toward actionable next steps.[cite:1]
